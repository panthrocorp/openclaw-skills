# Implementation Plan: Google Docs, Sheets, and Drive Comments for google-workspace

## Part 1: Design Decisions (Answering the 5 Key Questions)

### Question 1: Config model change for Drive

**Recommendation: (a) -- Promote `Drive` from `bool` to `DriveMode` with custom `UnmarshalJSON` for backwards compatibility.**

Rationale: Option (b) (`DriveComments bool`) fragments the Drive concept and will compound as more Drive write features are added. Option (c) (breaking change) is hostile to users who have deployed `config.json` files. Option (a) follows the Calendar precedent exactly and handles migration transparently.

The custom unmarshal logic is straightforward: if the JSON token is a boolean, map `true` to `"readonly"` and `false` to `"off"`. If it is a string, unmarshal normally. This is a 15-line method on a new `DriveMode` type.

### Question 2: Docs config

**Recommendation: Use `off`/`readonly`/`readwrite` as `DocsMode`, identical to `CalendarMode` and the new `DriveMode`.**

`true` as a value is inconsistent with every other multi-mode gate. The three types (`CalendarMode`, `DriveMode`, `DocsMode`) should share the exact same value space. In fact, all three can be a single generic `ServiceMode` type to reduce code duplication. I will define one `ServiceMode string` type with constants `Off`, `ReadOnly`, `ReadWrite` and redefine `CalendarMode` as an alias (or just replace it). However, since `CalendarMode` is already in use and the JSON values are stable (`"off"`, `"readonly"`, `"readwrite"`), renaming it is safe -- callers reference the constants, not the type name. I recommend the unified type approach.

### Question 3: Docs comments vs Drive comments

**Recommendation: Use the Drive API for ALL comments across all file types, including Docs.**

Rationale:
- The Drive Comments API (`drive.comments`) works uniformly on Docs, Sheets, Slides, PDFs, and all other Drive files.
- The Google Docs API does not have a comments endpoint at all -- comment management for Docs is done via the Drive API.
- Using a single API surface means one comment client, one set of tests, one set of commands.
- This eliminates the `docs comment` commands entirely. Comments always go through `drive comment`.

The issue lists `docs comment` and `docs comments list`, but these should be folded into the `drive comment` commands since the Drive API is the correct API for comments on any file type including Docs.

### Question 4: Scope escalation

**Recommendation: Use `drive.readonly` for `DriveMode="readonly"` and `drive` (full) for `DriveMode="readwrite"`. Do NOT use `drive.file`.**

Rationale:
- `drive.file` only grants access to files created by the app, which means it cannot comment on arbitrary user files.
- The `drive` scope is the only scope that allows commenting on files the user did not create via the app.
- The scope escalation is significant, but it is gated behind an explicit config change (`DriveMode="readwrite"`) plus re-authentication. The operator consciously opts in.
- The config UI should display a warning when Drive mode is changed to readwrite, similar to how Calendar already warns about re-authentication.

### Question 5: Docs edit command

**Recommendation: Expose two focused operations as flags on a single `docs edit` command, plus a raw JSON escape hatch.**

The three modes:
1. `docs edit --document-id <id> --insert-text "hello" --index 5` -- positional insert
2. `docs edit --document-id <id> --find "old text" --replace-with "new text"` -- find-and-replace-all
3. `docs edit --document-id <id> --requests-json '{"requests":[...]}'` -- raw batch update for advanced use

This avoids subcommand proliferation while covering the 90% use case (insert, find-replace) with simple flags. The raw JSON flag is the escape hatch for complex operations (delete ranges, formatting, etc.).

Separate `docs insert` / `docs replace` / `docs delete` subcommands are overkill for a CLI skill -- the agent will almost always use find-and-replace or insert-at-end.

```go
func (c *Config) UnmarshalJSON(data []byte) error {
    // Use a raw struct to avoid infinite recursion
    type raw struct {
        Gmail    bool            `json:"gmail"`
        Calendar ServiceMode     `json:"calendar"`
        Contacts bool            `json:"contacts"`
        Drive    json.RawMessage `json:"drive"`
        Docs     ServiceMode     `json:"docs"`
        Sheets   ServiceMode     `json:"sheets"`
    }
    var r raw
    if err := json.Unmarshal(data, &r); err != nil {
        return err
    }
    c.Gmail = r.Gmail
    c.Calendar = r.Calendar
    c.Contacts = r.Contacts
    c.Docs = r.Docs
    c.Sheets = r.Sheets

    // Handle Drive: bool (legacy) or string (new)
    if len(r.Drive) > 0 {
        if r.Drive[0] == 't' || r.Drive[0] == 'f' {
            var b bool
            if err := json.Unmarshal(r.Drive, &b); err != nil {
                return err
            }
            if b { c.Drive = ModeReadOnly } else { c.Drive = ModeOff }
        } else {
            var m ServiceMode
            if err := json.Unmarshal(r.Drive, &m); err != nil {
                return err
            }
            c.Drive = m
        }
    }
    return nil
}
```

## Part 4: Security Enforcement (Three Layers)

Following the existing pattern documented in CLAUDE.md:

### Layer 1: Code-level (compile time)

- `DocsClient` write methods (`InsertText`, `ReplaceAllText`, `BatchUpdate`) check `c.mode != config.ModeReadWrite` and return a descriptive error.
- `SheetsClient.WriteRange` checks `c.mode != config.ModeReadWrite` and returns an error.
- `DriveClient` comment methods (`CreateComment`, `ReplyToComment`) check `c.mode != config.ModeReadWrite` and return an error.
- Read-only clients for Docs/Sheets (when `mode == ModeReadOnly`) physically cannot call write methods because the mode guard fires first. This is the same pattern as `CalendarClient`.

### Layer 2: Config-level (runtime)

- CLI commands that perform writes check the mode BEFORE calling the client method. This is the existing Calendar pattern: `calendarCreateCmd` checks `mode != config.CalendarReadWrite` and calls `exitf`. New write commands for Docs, Sheets, and Drive comments will do the same.
- This provides a user-facing error message before the API call is even attempted.

### Layer 3: Google Cloud project (server-side)

- The operator must enable the Docs API and Sheets API in their Google Cloud project console.
- OAuth scopes are derived from config, so a readonly config never requests write scopes, meaning the token physically cannot perform write operations even if code bugs exist.
- CLAUDE.md should be updated to list the additional APIs: Google Docs API, Google Sheets API.

## Part 9: Potential Challenges

1. **Custom UnmarshalJSON interaction with `json.MarshalIndent`**: When saving, the new `Config` will marshal `Drive` as a string (`"readonly"`). Old binaries that read this config will fail to unmarshal `"drive": "readonly"` into `bool`. This is a one-way migration -- once the config is saved by the new binary, old binaries cannot read it. This is acceptable since the old binary would not have the new features anyway.

2. **Drive scope escalation requires re-auth**: When a user changes Drive from `readonly` to `readwrite`, they must run `auth login` again. The config set command already prints this warning. No additional logic is needed.

3. **Docs edit index calculation**: The `--index` flag for `InsertText` requires the user to know the character index in the document. This is fragile. # Security Audit Report: Google Workspace Skill (Docs, Sheets, Drive Comments)

**Audit date:** 2026-04-02
**Scope:** New Docs, Sheets, and Drive comments functionality added to the google-workspace OpenClaw skill
**Threat model:** Hostile agent running inside a Docker container on EC2, attempting to escalate privileges, bypass mode guards, exfiltrate data, or perform unauthorised writes

---

## FINDING 1 (CRITICAL): Drive scope escalation from `drive.readonly` to full `drive` scope

**File:** `/home/listellm/repos/panthrocorp/openclaw-skills/google-workspace/internal/config/config.go`, lines 194-199

When Drive is set to `readwrite`, `OAuthScopes()` requests `https://www.googleapis.com/auth/drive` (the full Drive scope). This is the broadest Google Drive scope available. It grants not just comment creation but also:

- File creation, deletion, and modification
- Permission changes (sharing files externally)
- Trash and permanent deletion
- Moving files between folders

The `DriveClient` struct does not expose file write/delete/share methods, so the code surface is limited. However, the **token itself** carries the full `drive` scope. A hostile agent that can read the token encryption key (`GOOGLE_WORKSPACE_TOKEN_KEY` environment variable) and the encrypted token file can construct its own HTTP requests against the Drive API outside this binary and perform any Drive operation.

In the hostile agent threat model, the encryption key is present as an environment variable in the container. The agent can trivially read it. The encrypted token is on disk. The agent can decrypt and use it directly.

**Recommendation:** Use a narrower scope. For Drive comments only, `https://www.googleapis.com/auth/drive.file` is insufficient (it only covers files the app created). However, consider whether `readwrite` mode for Drive should be restricted to comments-only by using a separate config flag (e.g. `drive_comments: true`) rather than escalating the entire Drive scope. At minimum, document prominently that `drive=readwrite` grants full Drive access at the token level, not just comment access.

**Risk:** A hostile agent with container access can use the full Drive scope token to delete files, exfiltrate data to external accounts by changing share permissions, or permanently destroy content.

## FINDING 2 (HIGH): `BatchUpdate` raw passthrough in DocsClient is an unrestricted write surface

**File:** `/home/listellm/repos/panthrocorp/openclaw-skills/google-workspace/internal/google/docs.go`, lines 103-115
**File:** `/home/listellm/repos/panthrocorp/openclaw-skills/google-workspace/cmd/docs.go`, lines 133-143

The `--requests-json` flag on `docs edit` accepts arbitrary Docs API batch update requests and passes them directly through to the Google Docs API. The Docs `BatchUpdate` method supports request types that go beyond text editing:

- `DeleteContentRange` (bulk content deletion)
- `InsertInlineImage` (inject images from external URLs, potential for phishing/tracking pixels)
- `InsertTable`, `DeleteTable`
- `CreateNamedRange`, `DeleteNamedRange`
- `ReplaceNamedRangeContent`
- `UpdateDocumentStyle`, `UpdateSectionStyle`

While the mode guard is correctly applied (`mode != ModeReadWrite`), the surface area is unbounded. A hostile agent in readwrite mode can perform any Docs API operation, not just the text insert/replace operations implied by the CLI's help text.

**Recommendation:** Either (a) restrict allowed request types in `BatchUpdate` to a safelist, or (b) remove the `--requests-json` passthrough entirely and only expose the specific operations the skill intends to support (insert text, find-and-replace). If the raw passthrough is intentional, document it as a known risk and ensure the SKILL.md makes this clear to operators.

## FINDING 3 (MEDIUM): `UnmarshalJSON` migration logic relies on first-byte heuristic

**File:** `/home/listellm/repos/panthrocorp/openclaw-skills/google-workspace/internal/config/config.go`, lines 71-89

The backwards-compatibility migration checks `r.Drive[0] == 't' || r.Drive[0] == 'f'` to detect legacy boolean values versus the new ServiceMode string. This works for the expected inputs (`true`, `false`, `"readonly"`, `"readwrite"`, `"off"`), but consider these edge cases:

- A JSON string `"false"` (the string value, not the boolean) starts with `"` so it correctly falls through to the string path, but it would fail `validateMode` later. This is safe.
- A raw JSON value like `0` or `1` (number) would not match `t` or `f`, would fall to the string unmarshal path, and would fail. This is safe but with a poor error message.
- If `r.Drive` is `null` (4 bytes, starts with `n`), `len(r.Drive) > 0` is true but neither branch matches, so it falls to string unmarshal of `null` which yields an empty ServiceMode, and Drive defaults to... nothing. The empty string is not caught here because there is no default-to-off for Drive when the string unmarshal succeeds with an empty string.

Wait, re-reading: the `json.Unmarshal(r.Drive, &m)` for a JSON `null` will set `m` to `""` (zero value), and then `c.Drive = ""`. This passes through to `Validate()` which calls `validateMode("drive", "")` and `""` does not match any of `off`, `readonly`, `readwrite`. So `Validate()` catches it. This is safe.

**Actual risk:** Low. The migration logic is sound for all realistic inputs and `Validate()` acts as a safety net. The only concern is that a hostile agent who can write to `config.json` could craft unusual JSON to probe for panics (e.g. empty `drive` array `[]`), but `json.Unmarshal` on `json.RawMessage` of `[]` into a bool or string would return an error, which is properly handled.

**Recommendation:** Add a unit test covering edge cases (`null`, `0`, `[]`, empty string) to document that these are handled. No code change required.

## FINDING 4 (MEDIUM): No runtime scope mismatch detection

**Files:** All `cmd/*.go` client constructors (`docsClient()`, `sheetsClient()`, `driveClient()`)

When an operator changes config from `readonly` to `readwrite` but does not re-authenticate, the token still carries the old (narrower) scopes. The binary does not detect this mismatch. The Google API will reject the request with a 403, but the error message from Google can be opaque.

More importantly in the hostile agent model: a hostile agent could run `config set --drive=readwrite` to change the config, but the token remains readonly-scoped. The agent cannot re-authenticate (requires human interaction via browser). This is actually a **defence** against scope escalation at runtime. The hostile agent can change the config file but cannot obtain a new token with broader scopes.

**Risk:** Low for security (the token is the enforcer). Medium for usability (confusing errors after config change).

**Recommendation:** On client construction, compare the scopes in the stored token against `cfg.OAuthScopes()`. If they differ, warn or refuse. This improves both UX and makes the defence-in-depth more explicit.

## FINDING 5 (MEDIUM): Docs and Sheets clients are constructable even in `ModeOff`

**File:** `/home/listellm/repos/panthrocorp/openclaw-skills/google-workspace/internal/google/docs.go`, line 25
**File:** `/home/listellm/repos/panthrocorp/openclaw-skills/google-workspace/internal/google/sheets.go`, line 25

**Recommendation:** Add a constructor guard: `NewDocsClient` and `NewSheetsClient` should return an error if `mode == ModeOff`. This matches the defence-in-depth principle. The `DriveClient` has the same gap.
