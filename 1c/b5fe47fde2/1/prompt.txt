```json
  "version": "0.1.0",
  "description": "Self-contained AWS S3 SDK bundle for OpenClaw agents",
  "main": "dist/index.cjs",
  "exports": {
    ".": {
      "require": "./dist/index.cjs"
    }
  },
  "files": [
    "dist/",
    "SKILL.md",
    "LICENSE"
```

```js
  sourcemap: false,     // Not needed for SDK re-export
  legalComments: "eof", // Preserve license comments at end of file
});
```

Key decisions:
- `platform: "node"` tells esbuild to treat Node built-ins (`http`, `https`, `stream`, etc.) as external -- they are not bundled. This is critical because the AWS SDK uses them.
- `format: "cjs"` because the gateway uses `require()`.
- `target: "node18"` -- conservative minimum, matching typical container base images.
- `minify: false` -- the bundle is not shipped to browsers; readability in the container aids debugging. The size difference is negligible for a server-side package.

### 13. Gotchas and Edge Cases

1. **`requires.npm` in SKILL.md is a new convention.** The existing skills use `requires.bins`. Confirm with the clawhub/openclaw team that `requires.npm` is a recognized field, or that unrecognized fields are ignored gracefully. If not supported, omit the machine-readable field and rely on the markdown install instructions.

2. **esbuild and Node built-ins.** With `platform: "node"`, esbuild marks Node built-ins as external. This is correct. However, if `@aws-sdk/client-s3` has any optional native dependencies (it should not, but verify), those would fail to bundle. The AWS SDK v3 is pure JavaScript, so this should not be an issue.

3. **Bundle size.** `@aws-sdk/client-s3` v3 is modular, but the full client with credential providers pulls in a significant dependency tree. Expect the bundle to be 1-3 MB. This is acceptable for a server-side package installed once into a container.

4. **`npm pack` tarball naming.** The default name is `panthrocorp-aws-s3-0.1.0.tgz` (npm lowercases and replaces `@`/`/` with the org prefix pattern). The `npm pack --json` approach in the release workflow handles this dynamically.

5. **`npm ci` requires `package-lock.json`.** The `aws-s3/` directory must commit a `package-lock.json`. Without it, `npm ci` fails. Generate it by running `npm install` during initial setup and commit it.

6. **release-please `node` release-type.** This release type bumps `version` in `package.json` and `package-lock.json` automatically. The `extra-files` YAML updater for `SKILL.md` handles the manifest version separately. Both will be updated in the release PR.

7. **No `prepack` in CI.** The `prepack` script in `package.json` runs `npm run build` before `npm pack`. In the release workflow, `npm run build` is called explicitly before `npm pack`, so the build runs twice. This is harmless (idempotent) but could be optimized by removing the explicit build call and relying on `prepack`. However, keeping the explicit call is clearer for CI readability.

8. **Credential provider bundling.** The default credential provider chain (`@aws-sdk/credential-provider-node`) is pulled in transitively by `S3Client`. This includes IMDS, ECS container credentials, environment variables, and shared credentials file providers. All of these are pure JS and will bundle correctly. Verify after the first build that instantiating `new S3Client({})` does not throw import errors.

9. **Container Node.js version.** The esbuild `target: "node18"` should match or be lower than the gateway container's Node version. If the gateway runs Node 20+, `node18` is safe. If it runs Node 16, adjust the target.

10. **Renaming the existing Go `build` job.** For clarity, consider renaming it to `build-go` when adding `build-npm`. This is optional but improves readability. The `gate` job's `needs` array would change to `[build-go, build-npm]`.

### LOW

**Change detection pattern for `ci.yml` is duplicated**

Both the `build` and `build-npm` jobs independently check for changes to `^\.github/workflows/ci\.yml$`. This means any change to `ci.yml` triggers both jobs, which is correct. However, if a second workflow file (e.g. `release-please.yml`) is modified without touching skill files, neither job triggers. This is probably fine for the current structure but worth noting as the repo grows.

**`gate` job accepts `skipped` as a passing state**

The gate loop checks only for `failure` or `cancelled`. A `skipped` result (which would occur if `build-npm` was removed or its `if:` condition failed at the job level) passes the gate silently. This matches the existing Go `build` job behaviour and is an accepted pattern here, but it means a misconfigured `if:` on a job will not block merges. Not a new risk introduced by this PR, but worth being aware of.

**Bundle size: 1.34 MB unminified**

The esbuild config sets `minify: false`. The AWS SDK is large but 1.34 MB unminified will be meaningfully larger than with minification enabled. For a gateway container that installs this once and caches it this is not a serious concern, but `minify: true` would reduce it significantly (likely 40-60%) with no functional change. Consider enabling it unless readable source in the container is explicitly needed for debugging.

**No `engines` field in `package.json`**

The esbuild target is `node22` but there is no `"engines": { "node": "&gt;=22" }` in `package.json`. A container running Node 18 or 20 would attempt to install and run the bundle without any upfront warning, and may encounter subtle incompatibilities if the output uses Node 22 APIs. Add the `engines` constraint.
